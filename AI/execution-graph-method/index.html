01 — The Core Idea

Turn architecture into a build system

Break the project into a DAG of small, testable tasks with explicit inputs and outputs. Execute in dependency order. Parallelize where safe.

A task isn’t “Build Workflow Engine.” A task is a bounded unit that has prerequisites, produces concrete outputs, and has acceptance checks.

02 — What an Execution Graph Is

Every node in the graph must be…

  • Prerequisite-aware — knows exactly what must be done first
  • Bounded — touches a known, limited set of files
  • Concrete — produces specific output artifacts
  • Verifiable — has acceptance checks that pass or fail
  • Assignable — has a clear owner (human or agent)

If a node doesn’t meet all five, it’s too vague. Break it down further.

03 — Graph Levels

Three zoom levels

Level 0
Master Roadmap The executive view. Lists every subsystem (Database, Backend, Frontend, DevOps…) and their coarse dependency order. Answers “which subsystems exist and which must finish before others can start.”
Level 1
Subsystem Graphs The tech-lead view. One graph file per subsystem. Task node IDs + names, internal dependency edges, cross-subsystem references, parallel groups, and rough sizing (~0.5–2 days per node). Does not contain file paths, line ranges, or verify commands.
Level 2
Execution Nodes The builder’s view. Full task spec — id, name, depends_on, inputs (exact file:line-range references), arch_rules, outputs, and verify commands. Everything needed to execute with zero ambiguity.

Each level is a zoom-in. Level 0 picks the subsystem. Level 1 picks the task. Level 2 gives you everything to execute it.

How they nest:

Level 0:  Master Roadmap
              │
              ├── Level 1:  database-graph.md     (task names + deps only)
              │                 ├── Level 2: P1-01 yaml block  (full spec)
              │                 ├── Level 2: P1-02 yaml block
              │                 └── ...
              ├── Level 1:  backend-graph.md
              │                 ├── Level 2: P1-06 yaml block
              │                 ├── Level 2: P1-07 yaml block
              │                 └── ...
              ├── Level 1:  frontend-graph.md
              └── ...
04 — Node Contract

What every node must declare

  • ID — stable, unique identifier
  • Description — what the task does
  • DependsOn — prerequisite node IDs
  • Inputs — docs, files, contracts to read
  • Outputs — files created or changed
  • Acceptance — tests, checks, endpoints, UI behaviors
  • Owner — agent role or person
  • Risk — P0 / P1 / P2
  • Rollback — how to revert safely

Every field is required. No ambiguity, no guessing. If info is missing, create a node that produces the missing artifact first.

05 — YAML Example

What a task spec looks like

File format: /plans/execution-graph.yaml

meta: project: Platform stack: backend: ".NET 10 Web API" frontend: "Angular + Angular Material" db: "SQL Server" rules: max_node_scope: "0.5-2 dev days" no_guessing: true if_unknown: "mark_requires_clarification" nodes: - id: G0-01 title: Establish repo build baseline owner: DevOpsAgent dependsOn: [] inputs: - "repo root" outputs: - "README-build.md" - "CI pipeline config" - "scripts/build.ps1 or scripts/build.sh" acceptance: - "Backend builds from clean checkout" - "Frontend builds from clean checkout" - "Tests (if any) run in CI" risk: P0 rollback: "Revert pipeline/script commits" - id: DB-01 title: Define core reference tables owner: DatabaseAgent dependsOn: [G0-01] inputs: # what to read - "docs/architecture/database-model.md" - "docs/architecture/domain-model.md" outputs: # what to produce - "db/migrations/0001_core_reference_tables.sql" - "db/schema/core_reference_tables.md" acceptance: # how to verify - "Migration runs on empty DB" - "Tables have PKs, FKs, required indexes" risk: P0 rollback: "Drop created tables OR down migration"
06 — Controller Prompt

The graph generation prompt

Feed this to an agent along with your architecture docs to produce the initial graph.

You are the Execution Graph Controller.

Inputs:
- docs/architecture/project-requirements.md
- docs/architecture/backend-architecture.md
- docs/architecture/frontend-architecture.md
- docs/architecture/database-model.md
- docs/architecture/system-risks.md

Task:
Generate /plans/execution-graph.yaml as a dependency
graph of executable nodes.

Rules:
- Each node: 0.5–2 dev days scope
- Required fields: id, title, owner, dependsOn,
  inputs, outputs, acceptance, risk, rollback
- No guessing — missing info = new prerequisite node
- Minimum coverage:
    10 foundation nodes (P0)
    30 domain slice nodes (DB→BE→FE chains)
    10 hardening nodes (tests, security, perf)
- Group by waves (Wave0..Wave5)
- Keep dependencies acyclic
- Prefer smaller nodes over larger ones

Output ONLY valid YAML.
Write to /plans/execution-graph.yaml.
07 — Building the Graph

Four steps to generate one

  • Map subsystem boundaries — Backend, Frontend, DB, Workflow, Files/Assets, Integrations, Security, DevOps/CI, Reporting
  • Create foundation nodes — build baseline, layering, migrations, auth baseline, logging/telemetry, OpenAPI baseline
  • Build vertical slices per domain — DB → Domain → API → UI → Tests → Permissions
  • Group into waves — Wave0 scaffolding; Wave1 reference data+identity; Wave2 core domain entities; Wave3 relationships+assets; Wave4 integrations+business rules; Wave5 workflows+reporting+hardening

Prefer smaller nodes over larger ones. A node scoped to 0.5–2 days is about right. If it feels bigger, split it.

08 — Parallelization

When parallel is safe

  • Different areas — db/ vs src/app/ vs src/Api/ can run simultaneously
  • Shared contracts first — interfaces and types must land before parallel consumers start
  • No overlapping files — two agents editing the same module at the same time is never safe

The graph’s dependency edges enforce this automatically. If two nodes don’t share an edge, they’re safe to parallelize.

09 — The Execution Loop

How it runs

Pick ready nodes
→
Assign by owner
→
Build outputs
→
Meet acceptance
→
Merge & repeat

A node is “ready” when all its dependencies are complete. Multiple ready nodes with no overlap can run in parallel.

Think make for software projects. Each target knows its prerequisites and its outputs. The graph engine just resolves the order.

10 — Why This Works

It eliminates the two things that kill AI-assisted builds

  • Ambiguity — every node has explicit inputs, outputs, and acceptance criteria. There’s nothing to interpret, nothing to assume. An agent either produces the artifact or it doesn’t.
  • Scope creep — nodes are bounded by design. An agent can’t wander into adjacent concerns because the graph doesn’t give it permission to. Dependencies enforce order; acceptance checks enforce quality.
  • Context overload — each node carries only what it needs. No 50-page prompt. No “here’s the whole codebase, figure it out.” Scoped context means better results from every agent, every time.

The graph isn’t a plan you hope works. It’s a machine you can run, inspect, and fix one node at a time.

11 — Start Here

Try it in five minutes

  • Pick one feature — something real but small. A login flow. A CRUD endpoint. A dashboard widget.
  • Decompose it into 3–5 nodes — each with inputs, outputs, and a way to verify it works.
  • Wire the dependencies — draw the edges. Which nodes can run in parallel? Which must wait?
  • Run the first node — hand it to an agent or build it yourself. Check the acceptance criteria. Move on.
  • Notice what happens — scope stays tight. Verification is binary. Progress is visible. That’s the method.

You don’t need to graph the whole application on day one. Start with one slice, prove it works, then expand. The graph grows with the project.
