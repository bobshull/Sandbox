01 — The Gap

You know the what and the how. You’re missing the bridge.

Article 1 told you to specialize your agents. Article 2 gave you the execution graph format — nodes with IDs, dependencies, inputs, outputs, acceptance criteria. But neither article answered the obvious next question: how do you actually get from “I have a project idea” to “I have a runnable graph?”

That’s the hard part. Not writing YAML. Not configuring agents. The hard part is decomposition — breaking a vague idea into concrete, bounded, dependency-ordered units of work that an agent (or a human) can execute without guessing.

This article walks through that process from start to finish, using one example the whole way through. No hand-waving. No “and then you decompose it” as if that’s a single step. We’ll show every decision, every split, every judgment call.

The graph is not the goal. Clarity is the goal. The graph is just what clarity looks like when you write it down.

02 — Start With What You Know

You don’t need a 50-page spec

Start with a plain-language description of what the app does. One paragraph. Maybe two. If you can’t describe it in a paragraph, you don’t understand it yet — and no amount of YAML will fix that.

Here’s our example. We’ll use it for the rest of this article:

“A task management app where teams can create projects, add tasks, assign them to team members, set due dates, and track progress. Users log in, see their dashboard of assigned tasks, and can comment on tasks. Project owners can view progress across all tasks in a project.”

That’s 50 words. It’s enough to start decomposing. You don’t need to know the tech stack. You don’t need wireframes. You don’t need a database schema. All of that comes out of the decomposition process — it doesn’t go into it.

What you do need:

  • Who uses it — team members, project owners (two roles, at minimum)
  • What they do — create projects, manage tasks, assign work, comment, view progress
  • What “done” looks like — a working app where those actions are possible

That’s your starting input. Everything else is derived.

03 — Extract the Domain

Pull out the nouns and the verbs

Read your description again. Underline the nouns — those are your entities. Underline the verbs — those are your operations. Don’t overthink it. You’re not designing a schema yet. You’re identifying the things the system cares about.

From our description:

User Logs in, has a role, gets assigned tasks
Team Groups users, owns projects
Project Contains tasks, has an owner, tracks progress
Task Has assignee, due date, status, belongs to project
Comment Attached to a task, written by a user

And the operations:

  • Auth — register, login, logout, manage sessions
  • Projects — create, list, view details, view progress
  • Tasks — create, edit, assign, set due date, change status, list, filter
  • Comments — create, list by task
  • Dashboard — view assigned tasks, view overdue tasks, view recent activity

Notice what we didn’t do: we didn’t pick a database. We didn’t decide on REST vs GraphQL. We didn’t argue about state management. That comes later. Right now we’re just naming the pieces.

If you find yourself designing implementation details at this stage, stop. You’re optimizing too early. Name the things first. Build them later.

04 — Draw Subsystem Boundaries

Group the work into areas that can be owned independently

Subsystem boundaries aren’t about technology. They’re about ownership. Each boundary defines a region of the codebase that one agent (or one developer) can work in without stepping on anyone else’s toes.

For our task management app, the subsystems are:

Database
Schema, migrations, seed data Everything that touches the relational DB directly. Table definitions, indexes, foreign keys, initial reference data. Owned by a database-focused agent.
Auth
Identity, sessions, permissions User registration, login, token management, role-based access control. A horizontal concern that every other subsystem depends on, so it needs to land early.
Backend API
REST endpoints, domain logic, validation The server-side code that exposes operations over HTTP. One set of endpoints per domain entity. Depends on database and auth.
Frontend
UI shell, pages, components, state The client-side application. App shell, routing, shared components, then feature-specific pages. Depends on API contracts being defined.
DevOps
Repo setup, CI, build scripts, deployment Everything that isn’t application code but is required to build, test, and ship it. Lands first because everything else depends on a working build.

Why these boundaries matter for parallelization: if two agents are working in different subsystems, they’re editing different files. No merge conflicts. No coordination overhead. The graph’s dependency edges handle sequencing automatically — the boundaries handle isolation.

A common mistake: making the boundaries too fine-grained. “Validation” is not a subsystem. “Error handling” is not a subsystem. Those are cross-cutting concerns that live inside other subsystems. We’ll handle those in slide 07.

05 — Identify the Foundation

What must exist before anything else can start?

Every graph has a bottom layer — nodes with no dependencies. These are your Wave 0 nodes. They set up the infrastructure that everything else builds on. Get them wrong and the entire graph stalls.

For our app, the foundation nodes are:

id: W0-01 title: Repository and build setup dependsOn: [] outputs: - Project repo with folder structure - Build scripts (backend + frontend) - CI pipeline config acceptance: - "Both backend and frontend build from clean checkout" id: W0-02 title: Database connection and migration framework dependsOn: [] outputs: - Database connection config - Migration runner setup - Empty initial migration acceptance: - "Migration runner connects to DB and runs empty migration" id: W0-03 title: Backend API skeleton dependsOn: [] outputs: - API project with health-check endpoint - Request/response pipeline configured - Error handling middleware acceptance: - "GET /health returns 200" id: W0-04 title: Frontend app scaffold dependsOn: [] outputs: - SPA framework initialized - Router configured with placeholder routes - Base layout component (nav, sidebar, content area) acceptance: - "App runs in dev mode, renders base layout"

Notice: all four nodes have dependsOn: []. They can all run in parallel. Four agents, four nodes, zero conflicts — because each one touches a different part of the codebase.

This is the first payoff of good subsystem boundaries. Your foundation layer is as wide as your boundary count.

The foundation isn’t glamorous. Nobody wants to set up CI pipelines and migration runners. But if these nodes fail, every downstream node is blocked. Treat them as P0.

06 — Build Vertical Slices

Take one entity and trace it through the entire stack

This is the core technique. Pick a domain entity — we’ll use Tasks — and trace it from database to UI. Each layer becomes a node. The dependencies between layers are your edges. One entity, one vertical slice, 4–6 nodes.

Five nodes. Clear chain of dependencies. Each node touches different files. Each has a concrete acceptance criterion that’s binary — it passes or it doesn’t.

Now repeat this for every entity: Users, Projects, Comments, Dashboard. Each one produces a similar chain. The slices share foundation nodes at the top and may share cross-cutting nodes (like auth) in the middle, but they’re otherwise independent.

Vertical slices are the highest-leverage decomposition technique. One slice proves the whole stack works end-to-end. After that, every additional slice is just more of the same pattern.

07 — Wire Cross-Cutting Concerns

Some things don’t get their own vertical slice

Auth, error handling, logging, and validation are horizontal. They cut across every slice. The temptation is to make them their own subsystem with a dozen nodes. Don’t. They’re support nodes that multiple slices depend on.

The key question: when does a cross-cutting concern become a node?

  • It produces a reusable artifact — middleware, a utility module, a shared config. Something other nodes reference as an input.
  • It has its own acceptance criteria — “auth middleware rejects expired tokens” is testable. “The app handles errors well” is not.
  • Multiple slices would duplicate the work — if three slices all need the same auth check, that’s a sign to extract it into its own node.

For our app, the cross-cutting nodes are:

id: BE-AUTH title: Auth middleware and token validation dependsOn: [W0-03, DB-USER] outputs: - "src/api/middleware/auth.{ext}" - "src/api/utils/token.{ext}" acceptance: - "Valid token passes middleware" - "Expired/invalid token returns 401" - "Role-based checks return 403 when unauthorized" id: FE-AUTH title: Login page, auth context, route guards dependsOn: [W0-04, BE-AUTH] outputs: - "src/app/pages/login.{ext}" - "src/app/context/auth.{ext}" - "src/app/guards/auth-guard.{ext}" acceptance: - "Login flow works end-to-end" - "Protected routes redirect to login when unauthenticated" id: BE-ERRORS title: Global error handling and validation pipeline dependsOn: [W0-03] outputs: - "src/api/middleware/error-handler.{ext}" - "src/api/middleware/validation.{ext}" acceptance: - "Unhandled exceptions return structured JSON error, not stack trace" - "Validation errors return 400 with field-level details"

These nodes sit between the foundation and the feature slices. Multiple slices depend on BE-AUTH and FE-AUTH, but those nodes only run once. The graph handles this naturally — a node with multiple dependents just means multiple downstream paths.

Watch for bottlenecks. If every single node in your graph depends on one cross-cutting node, that node is on the critical path. Make it small, make it reliable, and get it done first. A flaky auth node will block your entire project.

08 — Group Into Waves

Lay out the graph in execution order

Waves are just a visualization. They’re not a new concept — they’re the graph’s dependency layers, drawn horizontally. Wave 0 is everything with no dependencies. Wave 1 is everything that depends only on Wave 0. And so on.

Read this diagram top to bottom: everything in the same wave can run in parallel. An agent finishing Wave 0’s repo setup can immediately start on Wave 1’s DB migrations while another agent is still finishing the frontend scaffold.

The wave structure also tells you your critical path. The longest chain from top to bottom determines the minimum elapsed time. For our app, it’s roughly: repo setup → DB tables → auth → API endpoints → UI pages → integration tests. Six waves deep. Everything else happens in parallel alongside that chain.

Waves are a communication tool. Show them to stakeholders when they ask “what happens in what order?” Show the graph to agents when they need precise dependency info. Same data, different zoom level.

09 — Spot the Mistakes

Common decomposition errors (and how to fix them)

Every first-draft graph has problems. Here are the ones you’ll hit most often.

Nodes too big

Symptom: a node titled “Build the backend.” If you can’t estimate it in hours, it’s too big. Split it along entity boundaries or layer boundaries until each node is 0.5–2 days of work.

Bad: “Implement task management feature.” Good: “Create tasks table migration,” “Build task domain model,” “Add task REST endpoints,” “Build task list UI component.”

Nodes too small

Symptom: “Add status column to tasks table” as its own node. If a node takes 15 minutes and has no standalone value, merge it into its parent. The overhead of tracking a node should be less than the effort of doing it.

Missing dependencies

Symptom: an agent starts a node and immediately hits a blocker because something it needs doesn’t exist yet. Walk through each node’s inputs — does another node produce them? If not, either add a dependency edge or add a new node that produces the missing artifact.

Circular dependencies

Symptom: A depends on B, B depends on C, C depends on A. This means your boundaries are wrong. The fix is almost always to extract a shared interface or contract that both sides depend on, breaking the cycle.

# Circular: BE-TASK-API needs FE-TASK-UI for types, # FE-TASK-UI needs BE-TASK-API for endpoints # # Fix: extract a shared contract node id: CONTRACT-TASK title: Define Task API contract (request/response shapes) dependsOn: [W0-03] outputs: - "docs/api/task-contract.yaml" # OpenAPI or similar # Now both BE-TASK-API and FE-TASK-UI depend on CONTRACT-TASK # instead of on each other. Cycle broken.

Shared-file conflicts

Symptom: two nodes both list the same file as an output. When they run in parallel, one overwrites the other. Solutions: merge the nodes, sequence them with a dependency edge, or restructure so each node writes to different files.

Do a “conflict scan” before running: list every output file across all nodes. Any file appearing more than once is a conflict. Fix them before execution, not during.

11 — What Changes

Graphs aren’t static. Neither are requirements.

The graph you build on day one will not be the graph you finish with. Requirements shift, unknowns surface, nodes fail. That’s fine. The graph is designed to handle change — if you know the three operations.

Adding a node

A new requirement comes in: “We need email notifications when a task is assigned.” Create a new node. Wire its dependencies (it needs the task API and a mail service). Wire its dependents (nothing depends on it yet — it’s a leaf). Drop it into the appropriate wave. The rest of the graph is untouched.

id: BE-NOTIFY title: Email notification on task assignment dependsOn: [BE-TASK-API] outputs: - "src/api/services/notification.{ext}" - "src/api/templates/task-assigned.{ext}" acceptance: - "Assigning a task sends email to assignee" - "Email contains task title, project name, and link"

Re-wiring dependencies

You discover that the frontend auth guard actually needs user profile data that hasn’t been built yet. Add a dependency edge from FE-AUTH to BE-USER-API. This might push FE-AUTH to a later wave. That’s the graph telling you something true about your project — listen to it.

Splitting a node

A node is taking longer than expected because it’s actually two things. Split it. The original node’s dependents now depend on both new nodes (or just the one they actually need). Update the edges and keep going.

Never modify a completed node’s outputs. If downstream nodes already consumed those outputs, changing them breaks the chain. Instead, create a new node that produces the updated artifact and re-wire the dependents.

The graph is a living document. Treat it like code: version it, review changes, and don’t make edits that break downstream consumers. The graph discipline is the same as the code discipline.

12 — The Process, Summarized

A reusable decomposition checklist

This is the process. It works for a task management app, a billing system, a data pipeline, or anything else you can describe in a paragraph. The domain changes. The technique doesn’t.

  • Describe the app in plain language — one paragraph, no jargon, no implementation details. If you can’t do this, you’re not ready to build.
  • Extract entities and operations — nouns are entities, verbs are operations. List them. Don’t design anything yet.
  • Draw subsystem boundaries — group by ownership, not by technology. Each boundary is a region one agent can own without conflicts.
  • Identify foundation nodes — what must exist before anything else starts? These are your Wave 0 nodes with no dependencies.
  • Build vertical slices — trace each entity from DB to UI. Each layer is a node. Dependencies flow downward. This is the core technique.
  • Wire cross-cutting concerns — auth, error handling, validation. Extract them when multiple slices would duplicate the work.
  • Group into waves — sort by dependency depth. Everything at the same depth can run in parallel.
  • Scan for conflicts — check for shared output files, circular dependencies, nodes that are too big or too small. Fix them now.
  • Write the YAML — every node gets an ID, title, owner, dependsOn, inputs, outputs, acceptance, risk, and rollback. No optional fields.
  • Plan for change — requirements will shift. Know how to add nodes, re-wire edges, and split nodes without breaking the graph.

The graph is not the goal. Clarity is the goal. The graph is just what clarity looks like when you write it down.
